### Границы ответственности

#### Frontend (React + Bootstrap)
- Отображение доступных отчётов
- Форма запуска отчётов с параметрами
- Список запусков с пагинацией
- Статус генерации в реальном времени (с поллингом)
- Скачивание готовых файлов

**Не делает:**
- Не генерирует отчёты
- Не хранит состояние
- Не валидирует бизнес-логику

#### Backend API (FastAPI)
- REST API для управления отчётами
- Валидация параметров отчётов
- Создание записей о запусках
- Постановка задач в очередь Celery
- Отдача файлов на скачивание
- Пагинация и фильтрация списков

**Не делает:**
- Саму генерацию отчётов (делает через Worker)
- Хранение файлов (делает через Filesystem)
- Обработку очереди (делает Redis)

#### Worker (Celery)
- Асинхронное выполнение задач генерации
- Обновление статуса выполнения
- Обработка ошибок и retries
- Ограничение времени выполнения (time limit)

**Не делает:**
- HTTP обработку
- Прямое взаимодействие с клиентом

#### PostgreSQL
- Хранение метаданных о запусках
- Статусы, параметры, пути к файлам
- Временные метки

**Не хранит:**
- Сами файлы отчётов
- Очереди задач

#### Redis
- Брокер сообщений для Celery
- Хранение состояния задач (backend)

**Не хранит:**
- Бизнес-данные приложения

#### Filesystem (Docker Volume)
- Хранение сгенерированных XLSX файлов
- Разделяемый volume между Backend и Worker

### Поток данных

#### 1. Запуск отчёта
```
User → Frontend → POST /api/reports/{id}/run → Backend
                                                    ↓
                                              Создать запись в БД (status=pending)
                                                    ↓
                                              Celery task.delay() → Redis Queue
                                                    ↓
                                              Вернуть run_id клиенту (202 Accepted)
                                                    ↓
                                            Worker ← Redis Queue
                                                    ↓
                                              Обновить status=running
                                                    ↓
                                              Сгенерировать отчёт (OpenPyXL)
                                                    ↓
                                              Сохранить файл в /app/files
                                                    ↓
                                              Обновить status=completed, file_path
```

#### 2. Получение статуса
```
User → Frontend → GET /api/runs/{id} → Backend
                                            ↓
                                      SELECT из БД
                                            ↓
                                      Вернуть статус и метаданные
                                            ↓
Frontend: поллинг каждые 3 сек пока status != completed/failed
```

#### 3. Скачивание отчёта
```
User → Frontend → GET /api/runs/{id}/download → Backend
                                                    ↓
                                              Проверить status=completed
                                                    ↓
                                              Найти файл по file_path
                                                    ↓
                                              FileResponse (application/vnd.ms-excel)
```

## 2. Как добавить новый отчёт

### Пошаговая инструкция

#### Шаг 1: Создать генератор отчёта

Создайте файл `backend/reports/my_new_report.py`:

```python
from typing import Dict, Any, Optional
from openpyxl import Workbook
from reports.base import BaseReportGenerator


class MyNewReportGenerator(BaseReportGenerator):
    @property
    def id(self) -> str:
        return "my_new_report"  # Уникальный ID
    
    @property
    def name(self) -> str:
        return "My New Report"  # Отображаемое имя
    
    @property
    def description(self) -> str:
        return "Описание что делает этот отчёт"
    
    @property
    def params_schema(self) -> Optional[Dict[str, Any]]:
        # JSON Schema для параметров (опционально)
        return {
            "type": "object",
            "properties": {
                "days": {
                    "type": "integer",
                    "description": "Период в днях",
                    "default": 30,
                    "minimum": 1,
                    "maximum": 365
                }
            }
        }
    
    async def generate(self, params: Dict[str, Any], output_path: str) -> str:
        # Получить параметры
        days = params.get("days", 30)
        
        # Создать workbook
        wb = Workbook()
        ws = wb.active
        ws.title = "Report"
        
        # Заполнить данными
        # ... ваша логика генерации ...
        
        # Сохранить файл
        wb.save(output_path)
        return output_path
```

#### Шаг 2: Зарегистрировать отчёт

Откройте `backend/reports/registry.py` и добавьте импорт + регистрацию:

```python
from reports.my_new_report import MyNewReportGenerator  # <-- Добавить

REPORT_REGISTRY: Dict[str, Type[BaseReportGenerator]] = {
    SalesReportGenerator().id: SalesReportGenerator,
    APIUsageReportGenerator().id: APIUsageReportGenerator,
    MyNewReportGenerator().id: MyNewReportGenerator,  # <-- Добавить
}
```

#### Шаг 3: Проверить

1. Перезапустите сервисы: `docker-compose restart backend worker`
2. Откройте `GET /api/reports` — новый отчёт должен появиться в списке
3. Запустите отчёт через UI или `POST /api/reports/my_new_report/run`

### Конвенции

- ID отчёта: `snake_case`, уникален
- Файл: `backend/reports/{report_id}.py`
- Класс: `{ReportNameInCamelCase}Generator`
- Наследование от `BaseReportGenerator` обязательно
- Формат файла: по умолчанию `.xlsx` (OpenPyXL)
- Метод `generate()` должен быть async

### Что если нужен другой формат (PDF, CSV)?

Измените расширение в `output_path` и используйте соответствующую библиотеку:

```python
# Для PDF:
from reportlab.lib.pagesizes import letter
from reportlab.pdfgen import canvas
# ... генерация PDF ...
output_path = output_path.replace('.xlsx', '.pdf')

# Для CSV:
import csv
with open(output_path.replace('.xlsx', '.csv'), 'w') as f:
    writer = csv.writer(f)
    # ... запись данных ...
```

## 3. Принятые решения и альтернативы

### Решение 1: Celery vs ARQ vs FastAPI BackgroundTasks

**Рассматривали:**
- **FastAPI BackgroundTasks** — простейший вариант, встроен в FastAPI
- **ARQ** — async-native очередь на Redis, легче Celery
- **Celery** — индустриальный стандарт, зрелое решение

**Выбрали Celery, потому что:**
- Зрелая экосистема, огромное количество документации
- Встроенные механизмы retries, rate limiting, time limits
- Мониторинг через Flower (можно добавить одной командой)
- Легкое масштабирование workers
- Отдельный процесс для тяжёлых вычислений (не блокирует API)

**Минусы:**
- Дополнительный сервис в docker-compose
- Sync-код в Celery задачах (нужен asyncio.run())
- Overhead для прототипа

**Для продакшена:** рассмотрели бы ARQ для async-native подхода или KeepItSimple.

---

### Решение 2: PostgreSQL vs SQLite

**Рассматривали:**
- **SQLite** — нулевая конфигурация, один файл
- **PostgreSQL** — полноценная БД

**Выбрали PostgreSQL, потому что:**
- Реалистичнее для продакшена
- Лучшая конкурентность (SQLite имеет ограничения на write)
- JSONB для гибкого хранения параметров отчётов
- Генерация UUID на уровне БД
- Легкая миграция на продакшен (не нужно менять код)

**Минусы:**
- Дополнительный контейнер
- Дольше стартует

**Для прототипа:** SQLite был бы достаточен, но PostgreSQL показывает продакшен-готовность.

---

### Решение 3: React vs Server-Side Rendering (Jinja2)

**Рассматривали:**
- **Jinja2 + HTMX** — серверный рендеринг, минимум JS
- **React** — полноценный SPA
- **Vue.js** — альтернатива React

**Выбрали React, потому что:**
- Полировка UX (полиинг статуса, мгновенная навигация)
- Демонстрирует full-stack навыки
- Легко добавить real-time через WebSockets в будущем
- Bootstrap + React = быстрая разработка

**Минусы:**
- Overkill для CRUD интерфейса
- Node.js зависимость
- Дольше билд в Docker

**Для продакшена:** если команда бэкенд-ориентирована, HTMX был бы эффективнее.

---

### Решение 4: Абстрактный класс vs Конфигурация

**Рассматривали:**
- **Конфигурация (YAML/JSON)** — декларативное описание отчётов
- **Абстрактный класс** — каждый отчёт как Python класс
- **Plugin система** — динамическая загрузка из директории

**Выбрали абстрактный класс, потому что:**
- Type hints и IDE поддержка
- Легко тестировать
- Понятный контракт (BaseReportGenerator)
- Простая регистрация через dict
- Нет магии с импортами

**Минусы:**
- Нужно менять код для добавления отчёта
- Перезапуск сервисов

**Для продакшена:** рассмотрели бы YAML + template engine для нетехнических пользователей.

---

### Решение 5: Local File Storage vs S3/MinIO

**Рассматривали:**
- **Local filesystem** — файлы на диске
- **MinIO** — self-hosted S3-compatible
- **AWS S3** — облачное хранилище

**Выбрали локальный filesystem, потому что:**
- Прототип — не нужно ещё одно инфраструктурное сервис
- Docker volume обеспечивает персистентность
- Простота: путь в БД → файл на диске
- Нет сетевых задержек

**Минусы:**
- Не масштабируется горизонтально (нужен shared storage)
- Нет backup из коробки
- Limited access control

**Для продакшена:** S3/MinIO обязательно для масштабируемости и надёжности.

## 4. Что не сделали и почему

### Не реализовано

#### 1. Аутентификация и авторизация
**Почему:** Прототип фокусируется на архитектуре отчётов, не на безопасности.  
**Для продакшена:** JWT tokens, role-based access control (кто может запускать, кто скачивать).

#### 2. WebSocket для real-time обновлений статуса
**Почему:** Полиинг на фронтенде проще для прототипа.  
**Для продакшена:** WebSocket или Server-Sent Events для мгновенного обновления статуса без поллинга.

#### 3. Пагинация на уровне БД для больших объёмов
**Почему:** Реализована базовая пагинация (LIMIT/OFFSET), достаточно для прототипа.  
**Для продакшена:** Cursor-based пагинация для миллионов записей.

#### 4. Кэширование отчётов
**Почему:** Нет дедупликации — каждый запуск создаёт новый отчёт.  
**Для продакшена:** Проверка одинаковых параметров + временное окно → возврат готового отчёта без перегенерации.

#### 5. PDF с инфографикой
**Почему:** Заглушка в виде XLSX. Паттерн генерации тот же — просто другая библиотека.  
**Для продакшена:** ReportLab или WeasyPrint для PDF, Chart.js для графиков в PDF.

#### 6. Тесты
**Почему:** 4 часа — фокус на архитектуре и работающем коде.  
**Для продакшена:** Unit-тесты генераторов, integration-тесты API, e2e тесты UI.

#### 7. Мониторинг и логирование
**Почему:** Стандартные логи Celery/FastAPI достаточно для прототипа.  
**Для продакшена:** Prometheus + Grafana метрики, Sentry для ошибок, структурированное логирование.

#### 8. Rate limiting и квоты
**Почему:** Прототип предполагает доверенных пользователей.  
**Для продакшена:** Ограничение на количество запусков в час/день, приоритизация отчётов.

#### 9. Экспорт в несколько форматов одновременно
**Почему:** XLSX достаточно для демонстрации паттерна.  
**Для продакшена:** Генерация XLSX + PDF + CSV одним запуском, выбор формата в UI.

#### 10. Планировщик (cron)
**Почему:** Ручной запуск достаточно для прототипа.  
**Для продакшена:** Celery Beat для периодической генерации (ежедневно, еженедельно, ежемесячно).

## Что добавили бы для продакшена

### Критично
1. **Аутентификация/Авторизация** — JWT + RBAC
2. **S3 storage** — надёжное хранение файлов
3. **Миграции БД** — Alembic для управления схемой
4. **Тесты** — 80%+ покрытие
5. **CI/CD** — GitHub Actions с автоматическим деплоем

### Важно
6. **Мониторинг** — Prometheus, Grafana, алерты
7. **Логирование** — структурированное, централизованное
8. **Rate limiting** — защита от abuse
9. **Кэширование** — Redis кэш для частых запросов
10. **PDF экспорт** — инфографика и дашборды

### Желательно
11. **WebSockets** — real-time статусы
12. **Планировщик** — Celery Beat для периодических отчётов
13. **Уведомления** — email/Slack при завершении
14. **Шаблоны отчётов** — кастомизация дизайна
15. **Версионирование** — история изменений отчётов
